# time-decision-log

**素早い意思決定と日々の行動記録**のための、軽量なWebアプリケーション。

迷う時間を減らし、選択肢をクリックするだけで  
「今、何をしているか」を即座に記録できます。

アプリは自動でタイムスタンプを保存し、  
**日次・月次・四半期**のサマリーを 1 ページにまとめて表示します。

本アプリは **フロントエンド中心** の構成です。  
（フロントエンド：人が見る画面を作る部分）  

画面は自分で実装し、  
データの保存には **Supabase（外部サービス）** を利用しています。  
（バックエンド：データを保存する仕組み）

## 🧩 発生した主な課題一覧（Problem / Cause / Solution）

本プロジェクトでは、主に **4つの課題** に直面しました。

---

### ① 多言語化後、一部のUIだけが更新されなかった

**問題**  
一部のボタンや動的UIが、日本語に切り替わらず中国語のまま表示された。

**原因**  
表示内容がコード上の初期値ではなく、  
ブラウザの localStorage に保存された既存データを参照していたため。

**対応**  
サイトデータ（localStorage）を削除し、  
最新の初期値を再読み込みすることで解決した。

---

### ② GitHub Pages が反映されていないと誤認した

**問題**  
静的テキストは更新されているにもかかわらず、  
動的UIのみが更新されず、デプロイ失敗だと誤解した。

**原因**  
静的要素は HTML / CSS から直接描画されていた一方、  
動的要素は localStorage やバックエンドのデータを元に生成されていたため、  
参照しているデータソースが異なっていた。

**対応**  
UIごとに「どのレイヤー（HTML / JS / localStorage / DB）から生成されているか」を  
切り分けて確認するようにした。

---

### ③ フロントエンドとバックエンド接続時にエラーが発生した

**問題**  
フロントエンドからバックエンド（Supabase）へ接続した際、  
`401` および `42501` エラーが発生し、データの取得・登録ができなかった。

**原因**  
当初、`401` や `42501` が示す意味を理解できておらず、  
なぜリクエストが拒否されているのか判断できなかった。  
実際には、  
- 認証が必要な API に未認証の状態でアクセスしていた  
- 権限設定（RLS / GRANT）が不足していた  

ことが原因だった。

**対応**  
エラー内容をスクリーンショットで共有し、  
IT 経験者【義兄】に相談したところ、

- 「バックエンドの API はどこが提供しているのか？」
- 「この API はログインが必要なものではないか？」

という指摘を受けた。

その後、  
エラーコードの意味を調べ直し、  
認証状態および権限設定を確認・修正することで、  
正常に接続できるようになった。

---

### ④ ブラウザキャッシュにより変更が反映されないことがあった

**問題**  
コードを更新して GitHub に push した後も、  
画面が古い状態のまま表示されることがあった。

**原因**  
ブラウザに CSS / JavaScript のキャッシュが残っていたため。

**対応**  
ブラウザの開発メニューを有効化し、  
キャッシュの削除およびハードリロードを行った。

## ✅ 学び（Key Takeaways）

### ① 精細度（記録の粒度）

本プロジェクトを通して、  
**「思いついた段階から記録を残し、開発中も継続して記録すること」**の重要性を学びました。

具体的には、
- なぜこの機能を作ろうと考えたのか  
- 実装途中でどのような変更を行ったのか  
- どのような問題が発生し、当時どのように考えたのか  

といった内容を、できるだけ**細かく記録**するようにしました。

記録の粒度が高いほど、  
時間が経って詳細を忘れてしまっても、  
**振り返りや復盤をスムーズに行うことができ、  
同じ問題を繰り返さずに済む**と実感しました。

👉 **開発効率は、記録の精細度に大きく依存する**と考えています。

---

### ② 精准性（デバッグ時の正確さ）

デバッグにおいては、  
**やみくもに試行錯誤するのではなく、  
エラーメッセージを正しく読み取り、  
問題の発生箇所を特定する「正確さ」**が重要だと学びました。

特に、
- エラーがどのレイヤーで起きているのか  
  （フロントエンド／ブラウザ／データ／バックエンド）
- どの処理が原因で失敗しているのか  

を意識して切り分けることで、  
**問題箇所をピンポイントで修正でき、  
デバッグにかかる時間を大幅に短縮**できました。

👉 **デバッグの本質は、試行回数ではなく、  
正確な原因特定にある**と感じています。

---

### ③ 試行回数の蓄積による質的変化（量から質へ）

本プロジェクトを通して、  
**一定量の試行錯誤を積み重ねることで、理解の質が大きく変わる**ことを実感しました。

最初は、
- エラーの意味が分からない  
- 何を疑えばよいか判断できない  
- 手当たり次第に修正してしまう  

といった状態でしたが、  
**短期間で意識的に試行回数を増やし、  
その都度「なぜ失敗したのか」を振り返る**ことで、

- エラー内容から原因を推測できるようになる  
- 問題の切り分けが速くなる  
- 同じ種類のミスを繰り返さなくなる  

という変化が起きました。

👉 **試行錯誤の回数を増やしつつ、  
そこから経験を抽出することで、  
量的な経験が質的な理解へと転換される**と感じています。

## ✨ 機能（Features）

- ✅ **ワンクリック行動ログ**  
  あらかじめ用意された項目、または自分で追加した項目をクリックするだけで、
  活動を即時記録できます。

- 📊 **自動統計**
  - 日別サマリー
  - 月別サマリー
  - 四半期サマリー
  - カテゴリー別の回数・割合

- 🧾 **生ログ一覧（Raw log table）**
  - タイムスタンプ付きですべての記録を表示
  - 必要に応じて個別に削除可能

- 🗂 **カスタムカテゴリー**
  - 学習、コーディング、仕事、休憩など、自由に項目を追加可能
  - カテゴリーはローカルに保存されます

- 💾 **ローカルストレージ**
  - データはブラウザの `localStorage` に保存
  - バックエンド不要、アカウント不要、データ送信なし

- 📤 **CSVエクスポート**
  - 全記録をCSV形式で出力し、外部分析が可能

## 🧠 目的（Purpose）

このプロジェクトは、次のシンプルな問題を解決するために作られました。

> **考えすぎて時間を無駄にしない。  
> 行動し、記録し、振り返るだけ。**

行動記録のコストを極限まで下げることで、以下を実現します：
- 自己認識の向上
- 意思決定疲れの軽減
- 実際に時間がどう使われているかの可視化

※ 専門知識がない方でも使えることを意識して設計しています。

---

## ▶️ 使い方（How to Use）

1. ページを開きます  
2. 今の行動に近い項目をクリックします  
3. 自動で時間とログが記録されます

## 👩 テスター①（一般ユーザー視点）

**背景：家族（姉）**
- 小学校の数学教師として **約6年間の指導経験**
- IT専門職ではない立場
- 日常的に「分かりやすく伝える」ことを重視する職業背景

**主なフィードバック：**
- 画面だけでは **アプリの目的や使い方が分かりにくい**
- 用語や表現がやや抽象的で、一般ユーザーには理解しづらい
- 機能の意味が直感的に伝わらない

**改善方針：**
- 画面内に **分かりやすい説明文（UIガイド）** を追加
- 専門用語を避け、**誰でも理解できる表現**に変更
- 各機能に **補足説明（ツールチップ／説明エリア）** を追加
- 初回利用者向けの **簡単な操作説明（オンボーディング）** を検討

  - 編集機能  

  ユーザーが  
  **自由に管理できるUI** を目指します。

  ---
  
- **⑥ 状態の時間計測ロジック**

  状態ボタンを1回クリックすると  
  **「開始」状態**として記録します。

  次の状態をクリックするまで、  
  **同一状態が継続**されます。

  各状態の  
  **滞在時間を自動的に計測**します。

  👉 時間ベースの行動分析を可能にする設計です。

---

- **⑦ 期間指定と集計**

  当日／当月／当四半期の  
  **デフォルト期間表示**に対応します。

  **デフォルト表示：**
  - 当日：当日の24時間  
  - 当月：現在の月  
  - 当四半期：現在の四半期  

  **任意期間指定：**
  - 開始日（年月日）
  - 終了日（年月日）

---

- **⑧ 統計・可視化**

  行動データを  
  **回数・割合・時間**の観点で集計します。

  **カテゴリ別集計内容：**
  - 回数  
  - 回数比率（％）  
  - 合計時間  
  - 時間比率（％）

  **リアルタイム更新される  
  ツリー構造（階層表示）**により、  
  当日の行動状況を  
  **即時に把握できる可視化**を行います。

---

- **⑨ 生成AIを活用した対話型データ分析（AI活用視点）**

  ユーザーが自然言語で質問できる  
  **対話型検索UI** を想定しています。

  AIは、保存されている  
  **すべての行動履歴・時間データ** を参照し、  
  質問の意図を理解したうえで、  
  自動的に **集計・要約** を行います。

  **質問例：**
  - 「AI・ITの学習に最も時間を使った日はいつか」
  - 「今月、一番集中していたカテゴリは何か」
